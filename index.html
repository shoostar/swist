OVERVIEW

Mathematically Simple

Everything in SWIST+ is calculated on the fly, from colors, width, and font sizes, to gutters, columns, margins and padding. Now that CSS3 allows for the use of variables without a preprocessor, we can explicitly set variables, and then use them wherever we please, just like using the calc(); function, so that we can simply have the browser perform the math at run-time.

The codebase is a bit fat right now, but I'm working on chunking it down as much as possible. The size of the index.html file will more than likely grow as I add new examples, but I'm going to do my best to trim back the rest. I am also going to include some other shit, like form inputs and generic article/media layouts.

Universal*

While SWIST+ may not work on older browsers (honestly though, what does nowadays? And who really cares?), it should be perfectly compatible with all the latest and greatest browsers, regardless of the device. — *almost

GRID

Headache-free

SWIST+ is setup in a way that there are no complicated math equations for sorting out gridlet widths, gutters, padding, or other sizing adjustments. Rather, it's based on the premise that device widths and pixel densities that your audience views your site on are unknown.

To simplify a 12-column layout, this can be expressed as --columns: 12;, and then for each grid item (gridlet), simply assign a width that defines how many columns it spans. Pass a --width: variable along to that gridlet, and the rest is done by your browser. Setting --width: 3; means that gridlet spans 3 columns out of the 12 available (as defined by --columns: 12;).

Agnostic

With CSS' latest spec being pushed widely across all modern web rendering engines, we can now use native CSS variables on just about any device, and not break the internet. However, it's still highly recommended that you refactor the CSS and place it on top of a preprocessor to make things a bit easier, and to add other functions.

Responsive

Currently, SWIST+ has only 3 breakpoints on top of its default scale, but it's easy to add in your own. You need only to define how many columns a gridlet is to span, rather than having to calculate individual measurements for every container and breakpoint. Gridlets can be adjusted per-breakpoint by appending gridlet__(size)--# to that gridlet (or whatever naming convention you choose).

With the exception of your own CSS, everything should stack and un-stack cleanly, regardless of the breakpoint, gridlet size, or nesting. You only need to set at least one gridlet size, regardless of desired breakpoint.

Highly configurable

The first of the following examples is based on 8-columns, and while it wouldn't make sense to use so many oddball breakpoints for every single gridlet, but it just goes to show you how simple yet versatile this grid system can be. The second layout simply shows how you can not only extend the grid's container to the full-width of the screen, but adjust the column count on the fly as well. This example goes from having 8 columns to 9.

TYPE

Solid typeplate

I've taken the time to properly set up the font-scaling for this grid system's typeplate. Regardless of device or screensize, all text should scale proportionately to maintain the best readability across all screen sizes. I've also included a touch of JS that helps prevent widows in paragraph text.

While not difficult to change, I do advise taking a close look at the included notes if you're looking to alter the default/base font size and scaling. Everything has been carefully sized to align to a consistent baseline (regardless of font selector used), so attention will need to be paid to the calculations involved here.
